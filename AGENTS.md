# Agent Instructions for Ecoflow Cyber Deck

Welcome to the Ecoflow Cyber Deck repository. This codebase is split across three microcontroller platforms (ESP32, STM32F4, RP2040) and contains various rules, architectural paradigms, and custom tooling. When modifying this codebase, you **must** adhere to the following instructions.

## 1. Local RAG Indexing System
This repository uses a custom Retrieval-Augmented Generation (RAG) indexing system to help you navigate the codebase.
- You can search the index using `python rag_search.py "your query"`.
- **MANDATORY:** You must maintain the RAG index. Always run `python update_index.py` after you make code changes, before committing or pushing.

## 2. Microcontroller Architectures & Frameworks
- **EcoflowESP32**: Uses the `h2zero/NimBLE-Arduino` library version `^2.0.0` (upgraded from 1.4.1). Note that `NimBLEAdvertisedDeviceCallbacks` is replaced by `NimBLEScanCallbacks`, `onDisconnect` accepts an `int reason`, scan results receive `const NimBLEAdvertisedDevice*`, and `NimBLEScan::start` takes `(duration, is_continue)`.
- **EcoflowSTM32F4**: Built using PlatformIO targeting the `disco_f469ni` board.
- **EcoflowRP2040**: Uses the `earlephilhower` Arduino core for the Raspberry Pi Pico.

## 3. Critical Codebase Rules

### Duplicated Protocol Headers
- The `ecoflow_protocol.h` file is duplicated between `EcoflowSTM32F4/lib/EcoFlowComm/` and `EcoflowESP32/lib/EcoFlowComm/`.
- **Any modifications must be mirrored exactly in both locations** to ensure struct layouts align correctly for UART transmission.

### STM32 LVGL UI Memory Management
- In the EcoflowSTM32F4 LVGL implementation, repeatedly calling `lv_obj_add_style` on the same object without first explicitly removing the old style causes memory leaks (style nodes accumulate on the heap).
- **Always call `lv_obj_remove_style` before re-applying dynamically changing styles.**

### Data Parsing & Debug Logging
- **Code Generation:** The full debug dump implementation in `EcoflowDataParser.cpp` is generated by `python gen_logger.py`, which parses `.pb.h` files to ensure every field is logged. If you change protobuf files or need to update the dump, you must run `python gen_logger.py`.
- **State Structs:** `EcoflowDataParser` dump functions use comprehensive state structs defined in `EcoflowData.h` (e.g., `Delta3Data`), which expand to include every field from the protobuf definitions.
- **Unconditional Logging:** Debug dump functions must log all field values unconditionally, ignoring `has_` flags, to ensure zero values are captured and data dumps are complete.

### Other Important Implementation Details
- **ESP32 WebServer:** `WebServer::handleStatus` requires a `DynamicJsonDocument` size of at least 16KB (16384 bytes) to prevent dropping fields.
- **BLE Authentication & Connection:** During the `AUTHENTICATING` state, non-auth packets must be ignored rather than triggering an immediate failure. For Protocol V2 devices (e.g., Wave 2), a payload of `0x04` during authentication signifies success alongside `0x00`. To prevent immediate disconnection after auth, disable automatic ACKs (`shouldReply = false`) for incoming background telemetry and status packets.
- **Inter-chip UART:** The ESP32 and STM32F4 inter-chip UART communication uses a baud rate of 460800.

---
**Failure to follow these rules will result in compilation failures, communication breakdowns between microcontrollers, memory leaks, or missing data.**
